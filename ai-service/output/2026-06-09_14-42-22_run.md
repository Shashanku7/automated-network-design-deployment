## Phase 2: Network Topology

As a Senior Network Topology Architect, I have designed the following infrastructure. Based on the requirements, I have selected an **Aruba/HPE-centric design philosophy**, utilizing **VSX (Virtual Switching Extension)** for the core and **VSF (Virtual Switching Framework)** for the access layer to ensure a loop-free, highly available environment.

---

### 1. Topology Overview
**Type:** Hierarchical Three-Tier Campus Design (Core $\rightarrow$ Distribution $\rightarrow$ Access).
*   **Core Layer:** Collapsed Core/Distribution block located in the MDF (Main Distribution Frame).
*   **Distribution Layer:** Building-specific aggregation logic.
*   **Access Layer:** Floor-specific IDFs (Intermediate Distribution Frames) providing PoE+ for APs, VOIP, and IPTV.

### 2. Layer Breakdown & Sizing
The design maps the physical constraints of the PJ Building into logical blocks.

#### **Core/Distribution Block (MDF)**
*   **Hardware:** 2x High-capacity Chassis Switches (e.g., Aruba 8300 series).
*   **Logic:** Deployed as a **VSX Pair**. This provides a single logical entity for the access switches while maintaining a separate control plane for stability.

#### **Access Layer (IDFs per Floor)**
Sized based on device-to-user ratios (Students 2:1, Staff 1:1).

| Location | Load Calculation (Devices) | Switch Capacity Required | Logic |
| :--- | :--- | :--- | :--- |
| **Floor 1 (Admin)** | Staff(30) + Admin(2) + VOIP/IPTV $\approx$ 50 ports | 1x 48-port PoE+ Switch | VSF Stack |
| **Floor 2 (CSE)** | Students(400) + Staff(15) + Admin(1) $\approx$ 416 ports | 9x 48-port PoE+ Switches | VSF Stacks (2 stacks of 4+1) |
| **Floor 3 (ECE)** | Students(200) + Staff(1) $\approx$ 201 ports | 5x 48-port PoE+ Switches | VSF Stack |

---

### 3. High-Availability & Link Design
To eliminate single points of failure, the following redundancy mechanisms are implemented:

*   **Core Redundancy (VSX):** Two core switches acting as one. All distribution links are dual-homed.
*   **Access Redundancy (VSF):** Floor switches are stacked using VSF. If one member fails, the stack remains operational.
*   **Link Aggregation (LACP):** 
    *   **Uplinks:** Each Access Stack is connected to *both* Core switches via a **Multi-Chassis Link Aggregation (M-LAG/LACP)** bundle.
    *   **Speed:** 2 x 10GbE (SFP+) per stack, bundled for 20Gbps aggregate throughput.
*   **Failover:** VRRP (Virtual Router Redundancy Protocol) is used at the Core layer to provide a redundant default gateway for all VLANs.

### 5. Traffic Control & QoS Strategy
Following HPE/Aruba networking standards, traffic is prioritized at the ingress point (Access Layer) and honored through the Core.

*   **QoS Mapping:**
    *   **Voice (VLAN 100):** Strict Priority Queue. DSCP EF.
    *   **Video/IPTV (VLAN 110):** Guaranteed Bandwidth Queue. DSCP CS4.
    *   **Admin/Staff (VLAN 10, 21, 31):** Weighted Round Robin (WRR) Priority. DSCP AF21.
    *   **Students (VLAN 20, 30):** Best Effort. DSCP 0.
*   **Security ACLs:**
    *   `DENY Student_VLAN $\rightarrow$ Admin_VLAN` (Any/Any)
    *   `ALLOW Student_VLAN $\rightarrow$ Internet` (HTTP/S, DNS)
    *   `ALLOW Admin_VLAN $\rightarrow$ All_VLANs` (Full Management)

### 6. Summary of Failover Logic
1.  **Switch Failure:** If an Access Switch fails, VSF ensures the remaining switches in the stack maintain connectivity. If a Core Switch fails, the VSX peer takes over all traffic instantly with zero convergence time.
2.  **Link Failure:** If one fiber uplink in the LACP bundle is cut, traffic automatically redistributes to the remaining link without dropping the session.
3.  **Gateway Failure:** VRRP provides a virtual IP (VIP). If the Primary Core fails, the Backup Core assumes the VIP, ensuring the clients' default gateway remains reachable.

## Phase 3: Device Selection & BOM

Based on the approved topology and the technical specifications retrieved from the HPE Aruba Networking datasheets, here is the detailed Bill of Materials (BOM) and hardware selection for the PJ Building.

### Hardware Selection Logic

1.  **Core Layer (MDF):** 
    *   **Selection:** **CX 6400 Series**. 
    *   **Justification:** The design requires **VSX (Virtual Switching Extension)** for a high-availability, loop-free core. The CX 6400 explicitly supports VSX and provides the chassis-based scalability needed for a collapsed core/distribution block. It supports the required 10G/25G/100G uplinks to handle the aggregate traffic from three floors.

2.  **Access Layer (IDFs):**
    *   **Selection:** **CX 6200 Series** (specifically the **6200F** or **6200M** models).
    *   **Justification:** The design requires **VSF (Virtual Switching Framework)** stacking to ensure availability at the floor level.
        *   **CX 6100** was excluded because it does not support VSF stacking (only LACP/SmartLink).
        *   **CX 6200** supports VSF with up to 8 members and 40Gbps stacking bandwidth, fitting the requirements for the CSE and ECE floor stacks perfectly.
        *   **PoE Budget:** Selected the **740W** models to ensure sufficient power for the high density of VoIP phones, IPTV, and APs calculated in the load.

    *   **Core:** VSX implemented on CX 6400s $\rightarrow$ Separate control planes, single logical entity.
    *   **Access:** VSF implemented on CX 6200s $\rightarrow$ Single management plane for the floor stack.
*   **Connectivity:** 
    *   **Uplinks:** 2 x 10GbE SFP+ per stack, dual-homed to the VSX Core using LACP (M-LAG).
*   **PoE Performance:** 
    *   All access switches utilize the 740W power budget to support Class 4 PoE (up to 30W/port), accommodating the dense deployment of VoIP and IPTV.
*   **Throughput:** 
    *   Access switches provide up to 176 Gbps switching capacity, preventing bottlenecks for the high student population on Floor 2.
