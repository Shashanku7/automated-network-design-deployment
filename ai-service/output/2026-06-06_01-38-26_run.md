## Phase 2: Network Topology

As a Senior Network Topology Architect, I have designed the following infrastructure for the BMSCE Main Block. This design utilizes a **Collapsed Core/Distribution** model tailored for high-density academic environments, leveraging HPE Aruba networking standards (VSX/VSF) to ensure zero-downtime and high performance.

---

### 1. Topology Overview
**Topology Type:** Hierarchical Star (Three-Tier Architecture)
*   **Core/Distribution Layer:** A high-performance redundant pair of chassis switches located in the Main Server Room (Floor 1).
*   **Access Layer:** Floor-specific PoE+ switches providing connectivity to endpoints, WAPs, and VoIP devices.
*   **Logical Design:** Layer 3 routing occurs at the Core/Distribution layer; Layer 2 switching occurs at the Access layer.

---

### 2. Layer Breakdown & Sizing
Each floor is treated as a unique Access Block. To account for the requested **20% growth margin**, port densities are scaled upward.

| Floor | Dept | Total Endpoints (Current) | Design Port Capacity (incl. 20% Growth) | Recommended Access Switch Config |
| :--- | :--- | :--- | :--- | :--- |
| **1** | Admin | 40 | 48 | 1x 48-Port PoE+ Switch |
| **2** | 1st Year | 47 | 57 | 2x 48-Port PoE+ (or 1x 96-port stack) |
| **3** | E&C | 84 | 101 | 3x 48-Port PoE+ Stack |
| **4** | CS | 334 | 401 | 9x 48-Port PoE+ Stack (Multi-Chassis) |
| **5** | Info | 258 | 310 | 7x 48-Port PoE+ Stack |
| **6** | DS | 389 | 467 | 10x 48-Port PoE+ Stack (Multi-Chassis) |
| **7** | AI/ML | 320 | 384 | 8x 48-Port PoE+ Stack |

---

### 3. High Availability & Performance
*   **Core Layer (VSX):** Deployment of **Virtual Switching Extension (VSX)**. This provides a logically single switch instance across two physical chassis, allowing for active-active forwarding and sub-second failover.
*   **Access Layer (VSF):** Access switches on high-density floors (4, 5, 6, 7) will be configured in **Virtual Switching Framework (VSF)** stacks. This simplifies management and allows for cross-stack EtherChannel.
*   **LAG/LACP:** All uplinks from Access to Core will use **LACP (802.3ad)** bundles to provide both increased bandwidth and link-level redundancy.
*   **QoS Strategy:**
    *   **Voice (EF):** Highest priority for VoIP (DSCP 46).
    *   **Video (AF41):** Priority for IPTV and Video Conferencing.
    *   **Critical Data (AF21):** Admin and Staff traffic.
    *   **Best Effort (0):** Student Wi-Fi/General traffic.

---

### 4. Link Design & Physical Infrastructure
*   **Vertical Backbone (Riser):** 10GbE SFP+ OM4 Multimode Fiber running from the Main Server Room (Floor 1) to each floor's IDF (Intermediate Distribution Frame).
*   **Horizontal Cabling:** Cat6a shielded cabling from the floor switch to the wall jack/WAP (supports 10Gbps up to 100m).
*   **Uplink Redundancy:** Each floor stack will have **dual 10GbE uplinks**, each connecting to a different member of the VSX Core pair (Dual-Homing).

---

### 5. VLAN & Subnet Plan
To ensure security and reduce broadcast domains, a **Per-Floor, Per-Department** segmentation strategy is used. 
*   **Subnetting Logic:** Sized based on user counts + 20% growth.
*   **Isolation:** Inter-VLAN routing is managed at the Core via VRF or ACLs to ensure Students cannot access Admin/Staff segments.

### 6. Redundancy & Failover Summary
*   **Gateway Redundancy:** VRRP (Virtual Router Redundancy Protocol) or VSX Active-Gateway will be used. The "Virtual IP" serves as the default gateway for all floor VLANs.
*   **Loop Prevention:** MSTP (Multiple Spanning Tree Protocol) will be configured to prevent loops while allowing load balancing across VLANs.
*   **Power Redundancy:** All Core and Access switches must be equipped with Dual Hot-Swappable Power Supplies connected to independent UPS circuits.
*   **WAN Failover:** Dual-ISP entry points with BGP or Policy-Based Routing (PBR) for automatic failover if the primary circuit drops.

## Phase 3: Device Selection & BOM

Based on the approved topology and the provided Aruba CX datasheets, here is the hardware selection for the BMSCE Main Block. 

### Hardware Selection Strategy
*   **Core/Distribution:** The **Aruba CX 6400 series** (specifically the 6405 chassis) is selected to support the **VSX (Virtual Switching Extension)** requirement for high availability and zero-downtime.
*   **Access Layer:** The **Aruba CX 6300M series** is chosen for all floors. Unlike the 6100 or 6200F, the 6300M supports **VSF (Virtual Switching Framework)** stacking up to 10 members and features hot-swappable power supplies, meeting the "important" uptime requirement.
*   **Wi-Fi:** Based on the user counts and the requirement of ~25-30 users per AP, the quantity is calculated per floor.

### Technical Summary
*   **Redundancy:** Core is deployed as a **VSX Pair** (Active-Active). Access switches are deployed as **VSF Stacks**, allowing a single management IP per floor and sub-second failover.
*   **Port Density:** All access calculations include the **20% growth margin** as requested in the topology.
*   **Power:** The **CX 6300M** was selected over the 6300F because it allows for **Dual Hot-Swappable Power Supplies**, critical for maintaining uptime for VOIP and APs during PSU failure.
*   **Backbone:** 10GbE SFP+ OM4 fiber links are utilized to connect the floor IDFs to the Core.
